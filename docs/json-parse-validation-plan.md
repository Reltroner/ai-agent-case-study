# JSON Parse & Validation Plan — AI Sales Recommendation & Follow-Up Assistant for UMKM

## Purpose

This document defines the JSON parsing and validation plan for the AI Sales Recommendation & Follow-Up Assistant for UMKM.

The validation layer ensures that outputs generated by AI nodes and deterministic Code Nodes are:

- valid JSON
- structurally complete
- mathematically consistent
- grounded in the service catalog
- compatible with tracker schemas
- safe for human review
- routed correctly when errors occur
- auditable through raw and parsed output preservation

The validation layer must not silently accept malformed, inconsistent, or unsafe output.

---

## Architecture Position

The parsing and validation layer is positioned after structured output generation and before Google Sheet mapping.

```text
Manual Trigger / Google Form
-> Normalize Lead Input
-> Lead Qualification
-> Load Service Catalog
-> Deterministic Recommendation Code Node
-> AI Explanation / Follow-Up Drafting
-> JSON Parse & Validation Layer
-> Tracker Mapping
-> Google Sheet Append
-> Owner Notification
-> Human Review
```

The validation layer may also be used immediately after individual processing nodes:

```text
AI Lead Processing Node
-> Parse Lead Output
-> Validate Lead Output
```

```text
Recommendation Code Node
-> Validate Recommendation Output
```

```text
Follow-Up Drafting Node
-> Parse Combined Output
-> Validate Final Combined Output
```

# 3. Parsing Principles

The parser must follow these principles:

1. Preserve the raw node output.
2. Detect whether output is already an object.
3. Parse only when output is a string.
4. Remove harmless wrapping code fences when necessary.
5. Do not invent missing values during parsing.
6. Do not silently repair semantic errors.
7. Route invalid JSON to fallback handling.
8. Preserve parsing errors for audit.
9. Keep human review mandatory.
10. Never send customer-facing messages during an error path.

---

# 4. Accepted Input Formats

The parser may receive one of these formats.

## 4.1 Native JSON Object

Example:

```json
{
  "lead_id": "L001",
  "lead_category": "hot",
  "lead_score": 90
}
```

Action:

```text
Do not call JSON.parse.
Pass the object directly to structural validation.
```

---

## 4.2 JSON String

Example:

```text
{"lead_id":"L001","lead_category":"hot","lead_score":90}
```

Action:

```text
Trim whitespace.
Parse using JSON.parse.
Pass the parsed object to structural validation.
```

---

## 4.3 JSON Wrapped in Markdown Code Fence

Example:

````text
```json
{
  "lead_id": "L001"
}
```
`````

Action:

```text
Remove the opening and closing code fence.
Trim the remaining string.
Parse using JSON.parse.
Add a non-blocking warning:
markdown_code_fence_removed
```

---

## 4.4 Invalid JSON String

Example:

```text
{
  "lead_id": "L001",
  "lead_category": "hot",
}
```

Action:

```text
Do not guess the intended structure.
Do not silently remove the invalid comma.
Record the parsing error.
Route to fallback and manual review.
```

---

## 4.5 Empty Output

Examples:

```text
null
undefined
""
{}
```

Action:

```text
Treat as invalid or incomplete output.
Route to fallback and manual review.
```

An empty object may pass JSON parsing but must fail required-field validation.

# 5. Parsing Workflow

```text
Receive node output
-> preserve raw output
-> determine data type
-> normalize string wrapper
-> attempt JSON parsing
-> confirm parsed result is an object
-> proceed to schema validation
```

Detailed process:

```text
1. Read node output.
2. Store the original output as raw_output.
3. If output is a non-array object:
   -> use it directly.
4. If output is a string:
   -> trim whitespace.
   -> remove one outer Markdown code fence if present.
   -> call JSON.parse.
5. If parsing fails:
   -> create parsing error.
   -> route to fallback.
6. If parsed result is not an object:
   -> reject it.
7. Preserve parsed output as parsed_output.
8. Continue to structural validation.
```

# 7. Validation Stages

Validation should occur in ordered stages.

```text
Stage 1: Parsing validation
Stage 2: Required-field validation
Stage 3: Data-type validation
Stage 4: Allowed-enum validation
Stage 5: Mathematical consistency validation
Stage 6: Cross-field consistency validation
Stage 7: Catalog integrity validation
Stage 8: Safety validation
Stage 9: Tracker-mapping readiness
Stage 10: Final routing decision
```

A blocking failure at one stage must prevent the output from being treated as valid for normal processing.

# 12. Lead Score Validation

Formula:

```text
lead_score =
budget_clarity
+ timeline_urgency
+ service_clarity
+ buying_intent
```

Each score component must be:

```text
minimum = 0
maximum = 25
```

The total must be:

```text
minimum = 0
maximum = 100
```

Validation rules:

* `lead_score` must equal the score-breakdown total.
* A normal lead output must use a numeric score.
* A fallback output may use `null`.
* An unexplained score mismatch is a blocking error.

Error code:

```text
lead_score_mismatch
```



2. Offer name matches the ID.

Cross-validation rules:

* Recommendation lead ID must match lead-processing lead ID.
* Recommendation `lead_quality_fit` must match the lead category.

# 37. Array-to-Text Mapping

Google Sheets does not store JSON arrays as structured arrays.

Convert arrays using:

```text
array.join(", ")
```

Example:

```json
[
  "missing_budget",
  "missing_timeline"
]
```

Becomes:

```text
missing_budget, missing_timeline
```

Empty array becomes:

```text
empty string
```

Fields requiring conversion:

```text
missing_information
risk_flags
missing_recommendation_context
recommendation_risk_flags
```

# 44. Recommended Implementation Strategy

## MVP

Use one n8n Code Node to:

```text
parse strings
validate required fields
validate field types
validate enums
validate scores
validate catalog identity
validate human review
return validation envelope
```

## Later Improvement

Split into reusable modules:

```text

validate-recommendation-output.js
validate-catalog-integrity.js
validate-safety.js
build-fallback.js
```
